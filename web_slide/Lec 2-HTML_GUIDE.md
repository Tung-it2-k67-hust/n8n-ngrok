# DEVELOPER DECISION GUIDE: HTML

## SECTION 1: CORE MENTAL MODEL

### Core Definitions
- **HTML (HyperText Markup Language) (Ngôn ngữ đánh dấu siêu văn bản)**: Ngôn ngữ dùng để mô tả cấu trúc và nội dung tĩnh của trang web. HTML không phải là ngôn ngữ lập trình.
- **Tag (Thẻ)**: Ký tự đặc biệt dùng để đánh dấu loại nội dung và cấu trúc tài liệu.
- **Semantic HTML (HTML Ngữ nghĩa)**: Sử dụng các thẻ HTML mô tả chính xác ý nghĩa của nội dung (ví dụ: `<header>`, `<article>`) thay vì chỉ dùng `<div>` và `<span>`.

### Key Mental Models
1. **HTML là bộ xương (Skeleton)**: HTML chỉ định "cái gì" (what) và "đâu là quan trọng" (structure), không quan tâm "nó trông như thế nào" (style) hay "nó làm gì" (logic).
2. **Tree Structure (Cấu trúc cây)**: Trang web được trình duyệt hiểu như một cây DOM (Document Object Model) với node gốc là `<html>`.

### Why HTML Matters
- HTML là nền tảng duy nhất trình duyệt hiểu được. CSS và JS không thể tồn tại mà không có HTML.
- **SEO (Search Engine Optimization)**: Cấu trúc HTML tốt giúp công cụ tìm kiếm hiểu nội dung website.

## SECTION 3: ARCHITECTURE & RELATIONSHIPS

### HTML Document Hierarchy
Một file HTML hợp lệ phải tuân theo hierarchy sau:

```text
<!DOCTYPE html>
└── <html> (Root Element)
    ├── <head> (Metadata - Không hiển thị)
    │   ├── <meta> (Thông tin bảng mã, viewport)
    │   └── <title> (Tiêu đề tab trình duyệt)
    │
    └── <body> (Visible Content)
        ├── Semantic Structure (<header>, <nav>, <main>, <footer>)
        ├── Text Content (<p>, <h1>-<h6>)
        ├── Media (<img>, <video>)
        └── Interactive (<form>, <button>)
```

### The "Skeleton" Relationship
1. **HTML (Structure)**: Xác định vị trí các phần tử.
2. **CSS (Style)**: Bám vào HTML (ID/Class) để định kiểu.
3. **JS (Logic)**: Bám vào HTML (DOM Selectors) để thao tác.

*Quy tắc:* **"Chọn HTML trước, CSS theo sau, JS là tùy chọn"**.

## SECTION 4: CODE PATTERNS

### Pattern 1: Basic Valid HTML5 Document
**Khi nào dùng:** Luôn dùng khi bắt đầu file HTML mới. Đảm bảo trình duyệt hiểu đúng phiên bản.

**Tại sao đúng:** `<!DOCTYPE html>` kích hoạt chế độ "Standard Mode" (không phải "Quirks Mode").

```html
<!DOCTYPE html>
<html lang="vi">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Trang chủ</title>
</head>
<body>
    <h1>Xin chào thế giới</h1>
</body>
</html>
```

### Pattern 2: Semantic Article Structure
**Khi nào dùng:** Khi xây dựng blog, bài báo, hoặc nội dung văn bản có cấu trúc.

**Tại sao đúng:** Giúp công cụ đọc màn hình (Accessibility) và Google Bot hiểu được ý nghĩa các đoạn văn bản.

```html
<article>
    <header>
        <h1>HTML5 là gì?</h1>
        <time datetime="2023-10-01">01/10/2023</time>
    </header>
    
    <section>
        <h2>Giới thiệu</h2>
        <p>HTML5 là bản nâng cấp của HTML 4.</p>
    </section>

    <footer>
        <p>Tác giả: Senior Dev</p>
    </footer>
</article>
```

### Pattern 3: Robust HTML Form
**Khi nào dùng:** Khi cần người dùng nhập dữ liệu (Login, Register, Contact).

**Tại sao đúng:** Kết hợp `type` attribute và `required` attribute giúp trình duyệt tự xử lý validation cơ bản không cần JS.

```html
<form action="/submit-login" method="POST">
    <!-- Label giúp click vào chữ cũng mở ô input -->
    <label for="email">Email:</label>
    <input type="email" id="email" name="email" required placeholder="example@domain.com">
    
    <label for="password">Mật khẩu:</label>
    <input type="password" id="password" name="password" minlength="8" required>

    <button type="submit">Đăng nhập</button>
</form>
```

### Pattern 4: Responsive Image & Multimedia
**Khi nào dùng:** Hiển thị ảnh hoặc video.

**Tại sao đúng:** `alt` text bắt buộc cho SEO và người khuyết tật. `width/height` giúp tránh Layout Shift (tự động co giãn giao diện).

```html
<!-- Ảnh -->
<img src="logo.png" 
     alt="Logo của công ty TNHH" 
     width="200" 
     height="50">

<!-- Video không cần plugin Flash -->
<video controls width="400">
    <source src="movie.mp4" type="video/mp4">
    Trình duyệt của bạn không h hỗ trợ video.
</video>
```

## SECTION 5: ANTI-PATTERNS & WARNINGS

| Anti-Pattern (Mẫu phản kháng) | Why Dangerous (Tại sao nguy hiểm) |
| :--- | :--- |
| **Sử dụng thẻ cũ (Font, Center, B)** | Các thẻ như `<center>`, `<font>`, `<b>`, `<i>` đã bị loại bỏ. Trình duyệt vẫn hỗ trợ (backward compatible) nhưng code sẽ rác, khó bảo trì, và vi phạm quy tắc tách biệt nội dung (HTML) và định kiểu (CSS). |
| **Lạm dụng thẻ div (Div Soup)** | Dùng `<div>` cho mọi thứ (`<div class="header">`, `<div class="menu">`). **Nguy hiểm**: Mất đi ngữ nghĩa (Semantics), công cụ tìm kiếm không index đúng, người dùng screen reader không thể điều hướng. |
| **Thiếu `alt` attribute** | Để `<img src="...">` không có `alt`. **Nguy hiểm**: SEO tệ, người khuyết tật không biết ảnh này nói về gì. |
| **Input không có Label** | Dùng `<input>` không có `<label for="...">` (hoặc không bao bọc input). **Nguy hiểm**: Khó click (đặc biệt trên mobile), không thể sửa lỗi "truy cập bằng bàn phím" (Accessibility). |
| **Để ID trùng lặp** | Dùng hai thẻ có cùng `id="name"`. **Nguy hiểm**: CSS/JS chỉ tác động lên thẻ đầu tiên, gây lỗi logic rất khó debug. |

## SECTION 6: MASTER CHEAT SHEET

### Top 10 Rules
1. Luôn khai báo `<!DOCTYPE html>` ở đầu.
2. Đóng tất cả các thẻ (hoặc dùng cú pháp tự đóng `<img />`).
3. Sắp xếp thẻ `head` trước `body`.
4. Dùng **Semantic Tags** (header, main, footer) thay vì div tràn lan.
5. Luôn thêm `alt` vào `<img>`.
6. Dùng `<label>` cho mọi `<input>`.
7. Viết attribute `name` trong `<input>` nếu bạn muốn dữ liệu được gửi đi.
8. Dùng `type="email"`, `type="password"` thay vì text cho form.
9. Xác định `width` và `height` cho media để tránh giật giao diện (Layout Shift).
10. HTML chỉ để định cấu trúc, không dùng HTML để chỉnh sửa giao diện (dùng CSS).

### Decision Logic (If-Else)
*   **Nếu cần tạo nội dung văn bản?** -> Dùng `<p>`, `<h1>...<h6>`.
*   **Nếu cần tạo liên kết?** -> Dùng `<a href="...">`.
*   **Nếu cần form nhập liệu?** -> Dùng `<form>`, `<input>`, `<label>`.
*   **Nếu không có thẻ nào phù hợp?** -> Cân nhắc `<section>`, `<article>`, hoặc cuối cùng mới là `<div>`.

# DEVELOPER DECISION GUIDE: HTML Fundamentals (Lec 2)

## SECTION 1: CORE MENTAL MODEL

### Core Definitions
- **HTML Element (Phần tử HTML)**: The basic building block of a web page. It consists of opening tags, content, and closing tags (in most cases).
- **Block-level Element (Phần tử khối)**: An element that starts on a new line and takes up the full width available (e.g., `<div>`, `<p>`, `<h1>`).
- **Inline Element (Phần tử nội tuyến)**: An element that does not start on a new line and only takes up as much width as necessary (e.g., `<b>`, `<span>`).

### The Hierarchy Mental Model
Think of an HTML document as a tree (DOM Tree).
1.  **Root (`<html>`)**: The container for everything.
2.  **Metadata (`<head>`)**: Data *about* the document (not displayed).
3.  **Content (`<body>`)**: Data *visible* to the user.

### Why HTML Structure Matters
Unlike a text document where visual layout is everything, HTML relies on **Semantic Structure (Cấu trúc ngữ nghĩa)**. You separate *meaning* (Heading, Paragraph) from *style*. This allows browsers and screen readers to understand the content logic, which is why `<h1>` is always better than `<div>` with big font.

## SECTION 3: ARCHITECTURE & RELATIONSHIPS

### The DOM Hierarchy (Text Diagram)
```text
[HTML Document]
      |
      +--- <head> (Metadata: Not visible)
      |       |
      |       +--- <title> (Browser Tab Title)
      |       +--- <meta> (Charset, Description)
      |
      +--- <body> (Visible Content)
              |
              +--- <h1> (Block: Main Title)
              |
              +--- <p> (Block: Paragraph 1)
              |     |
              |     +--- <b> (Inline: Bold text inside para)
              |     +--- <span> (Inline: Highlight inside para)
              |
              +--- <div> (Block: Layout Wrapper)
              |     |
              |     +--- <p> (Block: Nested Paragraph)
              |
              +--- <hr> (Block: Separator)
```

### Element Flow Logic
1.  **Block Elements** always occupy a "row" of their own.
2.  **Inline Elements** share a "row" with siblings.
3.  **Nesting**: Block elements can contain Inline elements. Inline elements should **NOT** contain Block elements (though browsers try to fix it, it's invalid logic).

## SECTION 4: CODE PATTERNS

### Pattern 1: The Semantic Document Skeleton
**Khi nào dùng:** Khi bắt đầu mọi file HTML mới.
**Tại sao đúng:** Đảm bảo document type rõ ràng (HTML5) và cấu trúc cơ bản chuẩn xác để browser render đúng.

```html
<!DOCTYPE html>
<html lang="vi">
<head>
    <meta charset="UTF-8">
    <title>My First HTML Document</title>
</head>
<body>
    <!-- Nội dung sẽ nằm ở đây -->
    <h1>Hello World!</h1>
    <p>This is a paragraph.</p>
</body>
</html>
```

### Pattern 2: Paragraph with Internal Styling
**Khi nào dùng:** Khi cần in đậm hoặc highlight một phần nhỏ trong câu văn bản dài.
**Tại sao đúng:** Dùng `<b>` hoặc `<span>` (Inline) để không làm gián đoạn dòng文本.

```html
<body>
    <p>
        This paragraph contains a lot of lines
        but the browser <b>ignores it</b>.
        You can use <span style="color: red;">colors</span> too.
    </p>
</body>
```

### Pattern 3: Section Grouping with Div
**Khi nào dùng:** Khi cần bọc nhiều thẻ lại để xử lý layout (ví dụ: bọc 2 cột lại).
**Tại sao đúng:** `<div>` là khối generic linh hoạt nhất để tạo thành các block container.

```html
<body>
    <div id="header">
        <h1>Website Title</h1>
        <hr>
    </div>
    
    <div id="content">
        <p>Content goes here.</p>
    </div>
</body>
```

## SECTION 5: ANTI-PATTERNS & WARNINGS

### 1. Dùng `<br>` để tạo khoảng cách
**Lý do nguy hiểm:** Thẻ `<br>` (Line Break) chỉ dành cho việc xuống dòng thực sự (như địa chỉ thơ). Nếu bạn dùng `<br><br><br>` để tạo khoảng cách giữa các đoạn văn, bạn đang mix logic nội dung với presentation (khoảng cách).
**Đúng là:** Dùng Paragraph `<p>` (có margin mặc định) hoặc CSS để xử lý khoảng cách.

### 2. Bỏ qua `<title>` trong `<head>`
**Lý do nguy hiểm:** Browser sẽ hiển thị "Untitled Document" hoặc đường dẫn file, gây chuyên nghiệp kém và không có từ khóa SEO.

### 3. Dùng Block tag bên trong Inline tag
**Ví dụ:** `<span><div>Text</div></span>`
**Lý do nguy hiểm:** Cấu trúc này không hợp lệ (Invalid HTML). Browser sẽ cố gắng sửa lỗi nhưng hành vi có thể khác nhau giữa các trình duyệt, dễ gây lỗi layout khó lường.

## SECTION 6: MASTER CHEAT SHEET

### Top 10 Rules to Remember
1.  `<!DOCTYPE html>`: Bắt buộc ở dòng 1.
2.  `<html>`: Cha của mọi thứ.
3.  **Head**: Chứa info (title, meta) - không hiện thị.
4.  **Body**: Chứa content - có hiển thị.
5.  **Block**: Luôn xuống dòng (p, div, h1-h6).
6.  **Inline**: Không xuống dòng (b, i, span).
7.  **H1**: Chỉ 1 lần / trang.
8.  **Nesting**: Đóng thẻ theo thứ tự ngược lại (Mở `p` -> Mở `b` -> Đóng `b` -> Đóng `p`).
9.  **Attributes**: Nằm trong thẻ mở (e.g., `<html lang="vi">`).
10. **Whitespace**: Nhiều khoảng trống trong code HTML chỉ là 1 khoảng trắng khi hiển thị.

### Decision Logic (If-Else)
*   **IF** cần xuống dòng và chia tách section -> **Dùng Block (div, p, h)**.
*   **IF** chỉ cần style text trong dòng -> **Dùng Inline (span, b)**.
*   **IF** nội dung là bảng số liệu -> **Dùng table**.
*   **IF** nội dung là list -> **Dùng ul/ol**.
*   **IF** cần metadata -> **Dùng head**.

# DEVELOPER DECISION GUIDE: HTML Essentials

## SECTION 1: CORE MENTAL MODEL

### Semantic Structure & Data Collection
HTML is not just about visuals; it is about **Structure (Cấu trúc)** and **State Capture (Bắt trạng thái)**.
- **Text Appearance:** Defines visual hierarchy.
- **Navigation:** Links information spaces.
- **Data Presentation:** Tables display structured data.
- **Data Input:** Forms act as the bridge between user and server.

**Key Mental Model:** Think of HTML as the "Skeleton" (Bộ khung). CSS is the "Skin" (Da thịt), and JS is the "Muscles" (Cơ bắp). We focus here on the skeleton's ability to capture intent.

**Why this approach wins:** Native browser support means instant functionality without custom logic.

---

## SECTION 2: DECISION TABLES

### Text Formatting: Bold vs. Emphasis

| Use case (Tình huống sử dụng) | Should use (Nên dùng gì) | Why (Tại sao) | Common mistake (Sai lầm thường gặp) |
| :--- | :--- | :--- | :--- |
| Strict visual styling (e.g., product name) | **`<b>`** | **Presentational (Trang trí)**: Purely makes text bold, no semantic meaning for screen readers. | Using `<b>` for emphasis in a sentence, confusing screen readers. |
| Linguistic stress or semantic importance | **`<em>`** | **Emphasis (Nhấn mạnh)**: Tells screen readers to stress the word. Browsers default to italic, but meaning is key. | Using `<i>` or `<b>` instead, losing accessibility semantics. |
| Scientific math formulas | **`<sub>` / `<sup>`** | **Technical Notation (Ký hiệu)**: Standard for indices (H₂O) and exponents (x²). | Using tiny font size manually via CSS instead of semantic tag. |

### Lists: Order & Definition

| Use case (Tình huống sử dụng) | Should use (Nên dùng gì) | Why (Tại sao) | Common mistake (Sai lầm thường gặp) |
| :--- | :--- | :--- | :--- |
| Sequential steps or rankings | **`<ol>`** | **Ordered (Thứ tự)**: Browsers auto-number items. Sequence matters. | Using `<ul>` with manually typed "1." numbers. |
| Unrelated items or features | **`<ul>`** | **Unordered (Không thứ tự)**: Bullet points. Order does not matter. | Using `<br>` tags to separate items instead of list structure. |
| Dictionary-like terms & meanings | **`<dl>`** | **Definition (Định nghĩa)**: Pairs `<dt>` (Term) with `<dd>` (Definition). | Using a standard list to explain terms, losing machine readability. |

### Hyperlinks: Absolute vs. Relative

| Use case (Tình huống sử dụng) | Should use (Nên dùng gì) | Why (Tại sao) | Common mistake (Sai lầm thường gặp) |
| :--- | :--- | :--- | :--- |
| Linking to external websites | **Absolute URL** (`https://...`) | **External Reference (Tham chiếu ngoài)**: Full path ensures correct destination regardless of current file location. | Forgetting `https://` or `http://`, leading to broken local paths. |
| Linking within your own project | **Relative URL** (`images/pic.jpg`) | **Internal Portability (Di chuyển trong dự án)**: Keeps links valid even if you move the project folder. | Using absolute paths for local files, breaking deployment. |

### Images: Decorative vs. Informative

| Use case (Tình huống sử dụng) | Should use (Nên dùng gì) | Why (Tại sao) | Common mistake (Sai lầm thường gặp) |
| :--- | :--- | :--- | :--- |
| Image conveys critical information | **`alt="Description"`** | **Accessibility (Truy cập)**: Screen readers read this. Shows in place if image fails to load. | Leaving `alt=""` empty for important images (hidden by screen readers). |
| Purely decorative styling (borders, bg) | **`alt=""` (Empty)** | **Noise Reduction (Giảm nhiễu)**: Tells screen readers to ignore purely visual elements. | Adding descriptive text to decorative icons, cluttering audio output. |

### Forms: Text Input vs. Text Area

| Use case (Tình huống sử dụng) | Should use (Nên dùng gì) | Why (Tại sao) | Common mistake (Sai lầm thường gặp) |
| :--- | :--- | :--- | :--- |
| Short input (Name, Email, Code) | **`<input type="text">`** | **Linearity (Dòng đơn)**: Forces concise user input. | Using `<textarea>` for a 3-letter code, allowing unwanted line breaks. |
| Long descriptions (Comments, Address) | **`<textarea>`** | **Multi-line (Nhiều dòng)**: Supports scrolling and line breaks (`rows`/`cols`).** | Using `<input type="text">` with `width:100%`, hiding large text. |

### Forms: Dropdown vs. Radio Buttons

| Use case (Tình huống sử dụng) | Should use (Nên dùng gì) | Why (Tại sao) | Common mistake (Sai lầm thường gặp) |
| :--- | :--- | :--- | :--- |
| > 5 options or limited space | **`<select>`** | **Compactness (Tiết kiệm không gian)**: Collapses options until clicked. | Using radio buttons for >10 options, cluttering the UI. |
| < 5 options, immediate visibility | **`<input type="radio">`** | **Visibility (Hiển thị ngay)**: User sees all choices without clicking. | Hiding critical choices inside a dropdown, reducing conversion. |

### Hierarchy of Structure

```text
[ HTML Document ]
       |
       +-- [ Head ] (Metadata: Title, Charset, Links)
       |
       +-- [ Body ]
                |
                +-- [ Content Grouping ]
```

- **Cells (`<td>`)**: Must be inside a `<tr>`, which must be inside `<table>`.
- **Inputs (`<input>`)**: Functionality is largely defined by the parent `<form>` and attributes (`name`, `type`).

### Pattern 1: The Data Table
**When to use (Khi nào dùng):** Displaying structured records (e.g., student lists, inventories).

### Pattern 2: The Search Form
**When to use (Khi nào dùng):** Collecting simple data to send to a server.

**Why this pattern is correct (Tại sao đúng):** Explicit `method="POST"` prevents sensitive data from appearing in the URL. `name` attribute is required for the server to identify the data.

```html
<form action="/search" method="POST">

    <input type="text" id="query" name="search_query" placeholder="Enter keyword" required>
```

**When to use (Khi nào dùng):** User needs to write paragraphs (e.g., feedback, address, comments).

**Why this pattern is correct (Tại sao đúng):** `<textarea>` handles line wrapping automatically. `rows` sets initial visible height.

```html
<form action="/feedback" method="POST">

    <br>
    <textarea id="msg" name="feedback_text" rows="5" cols="50"></textarea>
    <br>
    <input type="submit" value="Send Feedback">
</form>
```

### Pattern 4: Selection Menu
**When to use (Khi nào dùng):** User must choose exactly one option from a long list (e.g., Country, Department).

**Why this pattern is correct (Tại sao đúng):** `<select>` keeps the UI clean. `name` attribute on `<select>` (not individual options) ensures the chosen value is sent.

```html
<form action="/register" method="POST">
    <label for="dept">Department:</label>
    <select id="dept" name="department_id">
        <option value="">--Please choose an option--</option>
        <option value="khmt">KHMT</option>
        <option value="vnn">Việt Nhật</option>
        <option value="httt">HTTT</option>
    </select>
    <br>
    <input type="submit" value="Register">
</form>
```

## SECTION 5: ANTI-PATTERNS & WARNINGS

### 1. Misusing Tags for Style Only
**Practice:** Using `<b>` to make a heading look bold because you don't want to use CSS.
**Danger (Nguy hiểm):** **Loss of Meaning (Mất nghĩa).** Screen readers won't emphasize it. Search engines won't recognize it as important. Always use `<h1>`-`<h6>` for headings, style with CSS later.

### 2. Empty or Missing `alt` Attributes
**Practice:** Leaving `<img src="...">` without the `alt` parameter.
**Danger (Nguy hiểm):** **Inaccessibility (Không thể truy cập).** Blind users hear "Image123.jpg" or nothing. If the image fails to load, the user has no idea what was there. Always provide `alt="description"`.

### 3. Nesting Lists Incorrectly
**Practice:** Putting `<li>` directly inside `<div>` or mixing `<ul>` and `<ol>` tags incorrectly.
**Danger (Nguy hiểm):** **Invalid HTML (HTML không hợp lệ).** Browsers may render unpredictably or break the list structure entirely. Strictly follow: `<ul>` -> `<li>` -> (nested `<ul>`).

### 4. Using Inputs without Labels
**Practice:** Just putting an `<input>` on the page without a `<label>` tag.
**Danger (Nguy hiểm):** **Usability Issues (Vấn đề trải nghiệm).** Users cannot click the text to focus the input. Screen readers cannot announce what the input is for. Always wrap input with `<label>` or use `for="id"`.

## SECTION 6: MASTER CHEAT SHEET

### Quick Reference Logic

**1. Need to capture text?**
- Short: `<input type="text">`
- Long: `<textarea>`
- Secret: `<input type="password">`

**2. Need user to choose?**
- One only, few options: `<input type="radio">`
- One only, many options: `<select>`
- Multiple options: `<input type="checkbox">`

**3. Need to organize data?**
- Visual grid: `<table>`
- Simple bullet points: `<ul>`
- Ranked steps: `<ol>`
- Term definitions: `<dl>`

**4. Need to link?**
- Internal file: `filename.html`
- External site: `https://url.com`

### Top 5 Rules to Remember
1.  **`<b>` is for BOLD (look), `<em>` is for EMPHASIS (voice).**
2.  **`<th>` is for Headers in tables, `<td>` is for Data.**
3.  **`<form>` needs `action` (where to go) and `method` (how to send).**
4.  **`<input>` needs `name` attribute to be sent to the server.**
5.  **`<img>` needs `alt` attribute always.**

# DEVELOPER DECISION GUIDE: HTML Forms & Multimedia (Lec 2)

## SECTION 1: CORE MENTAL MODEL

### Core Definitions
*   **Form (`<form>`)**: Container để thu thập dữ liệu người dùng và gửi về server. Hoạt động như một "bưu điện" số.
*   **Input (`<input>`)**: Thành phần cơ bản để nhập liệu, có nhiều loại (text, checkbox, radio, v.v.).
*   **Validation (Xác thực)**: Kiểm tra dữ liệu ngay trình duyệt (Client-side) trước khi gửi đi để đảm bảo tính chính xác.
*   **Attribute (Thuộc tính)**: Cấu hình behavior của thẻ HTML (ví dụ: `required`, `pattern`).

### Key Mental Model
Hãy coi Form là một **hệ thống ống nước**:
1.  **Input fields là các vòi nước**: Nơi dữ liệu đi vào.
2.  **Attributes là các lọc nước**: `required` hay `pattern` lọc tạp chất (dữ liệu sai định dạng) trước khi nước đi tiếp.
3.  **Form tag là đường ống chính**: Tập hợp tất cả nước lại.
4.  **Submit là bơm nước**: Đẩy dữ liệu qua đường ống (`action`) về nơi xử lý (`server`).

### Why HTML5 Form > JavaScript Form
*   **Tốc độ**: Validation diễn ra ở level browser (engine C++), nhanh hơn JS.
*   **Tự động**: Không cần viết lại logic basic (như check rỗng) mỗi lần làm form mới.
*   **Trải nghiệm người dùng (UX)**: Tự động hiện bàn phím số trên mobile khi gặp `<input type="number">`.

## SECTION 5: ANTI-PATTERNS & WARNINGS

1.  **Missing `name` Attribute:**
    *   **Danger:** `<input type="text" id="email">` (No `name`)
    *   **Why:** Dữ liệu nhập vào sẽ **KHÔNG** bao giờ được gửi lên server dù form có submit thành công. `id` chỉ dùng cho CSS/JS, `name` dùng cho server.

2.  **Incorrect Radio Grouping:**
    *   **Danger:** Dùng `id` khác nhau nhưng `name` khác nhau.
    *   **Why:** Người dùng có thể chọn tất cả các nút cùng lúc. Server sẽ nhận dữ liệu loạn xạ hoặc không biết xử lý option nào.

3.  **Using `GET` for Sensitive Data:**
    *   **Danger:** `<form action="/login" method="GET">`
    *   **Why:** Mật khẩu, Token sẽ hiển thị rõ ràng trên thanh URL, trong file log của server, và trong lịch sử trình duyệt.

4.  **Ignoring `required` and `pattern`:**
    *   **Danger:** Để trống validation HTML, chỉ dùng JS check khi submit.
    *   **Why:** Tăng tải nặng cho server (phải reject request sai) và trải nghiệm người dùng kém (phải đợi server response mới biết lỗi).

### Top 10 Rules
1.  **Checkbox**: Chọn nhiều, không cần `name` chung.
2.  **Radio**: Chọn 1, **bắt buộc** `name` chung.
3.  **Submit**: Phải ở trong thẻ `<form>`.
4.  **Reset**: Hồi phục giá trị ban đầu, không xóa trắng.
5.  **Button**: `type="button"` = Không làm gì, phải có JS.
6.  **GET**: Thấy rõ trong URL, dùng cho Tìm kiếm.
7.  **POST**: ẩn trong Body, dùng cho Đăng nhập/Upload.
8.  **Attribute**: `required` bắt buộc, `pattern` định dạng regex.
9.  **Label**: Luôn gắn với Input qua `for` & `id` để dễ click.
10. **Name**: Thiếu `name` = Thiếu dữ liệu gửi đi.

### Decision Logic (If-Else)
*   **IF** người dùng chọn > 1 option **THEN** dùng `Checkbox`.
*   **IF** người dùng chỉ được chọn 1 option **THEN** dùng `Radio` (cùng `name`).
*   **IF** dữ liệu cần hiển thị trên URL **THEN** dùng `method="GET"`.
*   **IF** dữ liệu là mật khẩu/file **THEN** dùng `method="POST"`.
*   **IF** nút bấm cần làm gì đó ngoài submit **THEN** dùng `type="button"` + JS.

# DEVELOPER DECISION GUIDE: HTML Multimedia & Semantic Structure

---

## SECTION 1: CORE MENTAL MODEL

### Core Definitions
- **Multimedia (Đa phương tiện)**: Content types combining text, audio, images, and video. In HTML, handled via native `<img>`, `<audio>`, `<video>`, and `<iframe>` tags without third-party plugins.
- **Semantic HTML (HTML Ngữ nghĩa)**: Using HTML tags to describe the **meaning** of content, not just its appearance. It provides context to browsers, search engines, and assistive technologies.

### Key Mental Models
1.  **Browser as a Structured Document Parser**: Browsers read HTML tags to build a DOM tree. Semantic tags act as landmarks (`<header>`, `<main>`, `<footer>`), allowing screen readers and search engines to navigate the page structure efficiently.
2.  **Separation of Concerns**: HTML defines *structure and meaning* (Semantics). CSS defines *visual presentation* (Layout/Color). Do not use HTML for styling (e.g., `<center>`, `<font>`).
3.  **SEO as Machine Reading**: Search engines (Google) cannot "see" images or understand unstructured text. Semantic tags and `alt` attributes provide the metadata bots need to index content correctly.

### Why This Approach Is Better
- **Accessibility**: Screen readers use semantic tags to announce page sections (e.g., "Navigation", "Article"), allowing visually impaired users to skip content.
- **Future Proofing**: Styling logic lives in CSS. If design changes, you update CSS, not the HTML structure.
- **Performance**: Native `<video>` and `<audio>` tags are optimized by the browser engine, whereas Flash or heavy JS libraries are resource-intensive.

---

## SECTION 2: DECISION TABLES

### Table 1: Structure - `<div>` vs Semantic Tags
| Use case (Tình huống sử dụng) | Should use (Nên dùng gì) | Why (Tại sao) | Common mistake (Sai lầm thường gặp) |
| :--- | :--- | :--- | :--- |
| **Main Content Wrapper** | `<main>` | **Landmark Role**: Tells bots/screen readers "This is the unique content of the page." (Distinct from `<body>`). | Using `<div id="main">` or `<div class="main">`. No semantic meaning is conveyed. |
| **Independent Content (e.g., News, Blog)** | `<article>` | **Independence**: Content remains meaningful if distributed to other sites (RSS, syndication). | Wrapping everything in `<section>` just to get a container. |
| **Grouping Related Content** | `<section>` | **Thematic Grouping**: Groups content by a specific theme/topic, usually with a heading. | Using `<section>` for styling purposes (e.g., just to add a border). Use `<div>` instead. |
| **Tangential Content (e.g., Sidebar)** | `<aside>` | **Indirect Relevance**: Content separate from the main flow (ads, author bio, TOC). | Placing critical navigation or content inside `<aside>`. It should not be essential to the main narrative. |

| **Emphasis / Stress (Voice Tone)** | `<em>` | **Vocal Stress**: Indicates emphasis when read aloud. Changes the meaning of the sentence. | Using `<i>` for foreign words. Use `<dfn>` or keep `<i>` but add `lang` attributes for better semantics. |
| **Stylistic Offset (No Stress)** | `<i>` | **Alternative Voice**: Technical terms, thoughts, ship names. Visual only. | |

### Table 3: Multimedia - `<img>` vs `<iframe>`
| Use case (Tình huống sử dụng) | Should use (Nên dùng gì) | Why (Tại sao) | Common mistake (Sai lầm thường gặp) |
| :--- | :--- | :--- | :--- |
| **Static Visual Content** | `<img>` | **Direct Asset**: Loads a specific file (JPG, PNG, SVG). Supports `alt` and `width/height` natively. | Using `<div>` with `background-image`. This hides the image from screen readers and SEO. |

| **Audio/Video Playback** | `<audio>` / `<video>` | **Native Controls**: Provides browser-built play/pause UI and accessibility hooks. No JS required. | Linking directly to media files (`<a href="song.mp3">`). The browser won't play it inline; it will download or open a new tab. |

## SECTION 3: ARCHITECTURE & RELATIONSHIPS

### HTML Semantic Hierarchy
HTML documents are structured like a book. Search engines parse this hierarchy to understand topic weight.

```text
[Root]
  └── <html>
      ├── <head> (Metadata only)
      │     ├── <title> (SEO Title)
      │     └── <meta description> (SEO Snippet)
      │
      └── <body> (Visible Content)
            ├── <header> (Site Header / Logo)
            │
            ├── <nav> (Main Navigation Links)
            │
            ├── <main> (Unique Page Content)
            │     ├── <article> (Independent Post/Story)
            │     │     ├── <header> (Post Title/Meta)
            │     │     ├── <section> (Chapter/Topic)
            │     │     └── <footer> (Post Author/Date)
            │     │
            │     └── <section> (Comments/Related Content)
            │
            ├── <aside> (Sidebar, Ads, Related Links)
            │
            └── <footer> (Copyright, Contact Info)
```

## SECTION 4: CODE PATTERNS (READY TO USE)

### Pattern 1: The Semantic Article Structure
**When to use:** Blog posts, news items, user comments (any self-contained content).
**Why this pattern is correct:** It isolates content for SEO (Google loves `<article>`) and allows users to share a specific "piece" of your page.

```html
<!-- Complete Semantic Article Example -->
<article>
  <header>
    <h1>The Future of Web Development</h1>
    <p>Published on <time datetime="2023-10-25">October 25, 2023</time> by <span class="author">Jane Doe</span>.</p>
  </header>

  <section>
    <h2>Introduction</h2>

  </section>

  </figure>

  <footer>
    <p><em>End of article. View comments below.</em></p>
  </footer>

```

### Pattern 2: Native Media Embedding
**When to use:** You need to play audio or video directly on the page.
**Why this pattern is correct:** Uses native browser controls (accessible via keyboard), supports `preload` strategies for performance, and requires no external JS libraries.

```html
<!-- Audio Example -->

  <source src="podcast.mp3" type="audio/mpeg">
  <source src="podcast.ogg" type="audio/ogg">

<!-- Video Example with Fallback -->
<video controls width="640" height="360" poster="thumbnail.jpg" preload="metadata">

  <p>
    Your browser doesn't support HTML5 video. 
    <a href="tutorial.mp4">Download the video file</a>.
  </p>
</video>
```

### Pattern 3: SEO & Accessibility Metadata
**When to use:** Every page head.
**Why this pattern is correct:** `title` and `meta description` control your search result snippet. `alt` text describes images for Google Image Search and blind users.

```html
  <meta name="viewport" content="width=device-width, initial-scale=1.0">

    src="rog-laptop.jpg"

    width="800" height="450"
  >
</body>
```

## SECTION 5: ANTI-PATTERNS & WARNINGS

### 1. Using `<div>` and `<span>` for Everything
**Danger:** Creating a "Div Soup". If you wrap a navigation menu in `<div class="nav">`, you are lying to the browser. It sees a generic box, not a list of links.
**Why dangerous:** Screen readers cannot find the navigation to let users skip it. Search engines cannot identify the main content area easily.

### 2. Missing `alt` Attributes on `<img>`
**Danger:** `<img src="cat.jpg">`
**Why dangerous:** Blind users hear "Image" or the file name (cat.jpg). Google Image Search cannot index the image. It is a huge accessibility failure.

### 3. Using Deprecated Styling Tags
**Danger:** Using `<center>`, `<font>`, `<strike>`, or `<u>`.
**Why dangerous:** These are obsolete. Centering text with `<center>` is rigid. If you want to center text, use CSS `text-align: center;`. Mixing style into HTML makes maintenance a nightmare.

### 4. Ignoring Heading Hierarchy
**Danger:** Using `<h1>` for the logo, `<h2>` for the footer, and `<h3>` for the main article title.
**Why dangerous:** Search engines use headings to build a Table of Contents for the page. Skipped levels (e.g., `<h1>` to `<h3>`) confuse the algorithm about the importance of content.

## SECTION 6: MASTER CHEAT SHEET

### Decision Logic (If-Else)
- **Is the content independent and distributable?** -> Use `<article>`.
- **Is the content grouped by a specific theme/topic?** -> Use `<section>`.
- **Is the content tangential (sidebar, callout)?** -> Use `<aside>`.
- **Is it a list of links?** -> Use `<nav>` inside a `<ul>`.
- **Is it a visual element that needs description?** -> Use `<img>` with `alt`.
- **Is it external web content (maps, ads)?** -> Use `<iframe>`.

### Top 10 Things to Remember
1.  **`<main>`**: Only one per page.
2.  **`<article>`**: Self-contained content.
3.  **`<section>`**: Thematic grouping (usually has a heading).
4.  **`<aside>`**: Tangential content.
5.  **`<strong>`**: Importance (Voice: LOUD).
6.  **`<em>`**: Emphasis (Voice: Stress).
7.  **`<img alt="...">`**: Mandatory for SEO and Accessibility.
8.  **`<video controls>`**: Native UI is better than JS custom players.
9.  **No `<center>`**: Use CSS (`text-align: center`).
10. **`<time datetime="...">`**: Machine-readable dates for SEO.

## SECTION 1: CORE MENTAL MODEL

### SEO (Tối ưu hóa công cụ tìm kiếm)
Mental Model: HTML không chỉ là giao diện cho người dùng mà còn là API cho các con bot (Search Engine Bots). Bạn cần "nói" với bot chính xác nội dung trang web là gì, cấu trúc ra sao, và mức độ quan trọng của từng phần.

Mental Model: Thay vì chỉ dùng `<div>` cho mọi thứ, hãy chọn tag mô tả chính xác nội dung. `<article>` khác với `<section>`, `<nav>` khác với `<aside>`.

## SECTION 5: ANTI-PATTERNS & WARNINGS

1.  **Dùng `<div>` thay cho `<h1>...</h6>` và CSS để làm to chữ.**
    *   **Nguy hiểm:** Mất cấu trúc ngữ nghĩa (Semantic Structure). Bot tìm kiếm không hiểu đâu là tiêu đề chính, giảm thứ hạng SEO.

2.  **Title tag quá dài hoặc trùng lặp.**
    *   **Nguy hiểm:** Google sẽ cắt Title (truncation) hoặc coi nó là spam. Title là yếu tố SEO On-page quan trọng nhất.

3.  **Sử dụng hình ảnh không có thuộc tính `alt` (hoặc `alt` rỗng).**
    *   **Nguy hiểm:** Bot không thể hiểu nội dung hình ảnh, người dùng khiếm thị không biết hình ảnh này nói về gì. `alt` là bắt buộc.

4.  **Lạm dụng `<br>` để tạo khoảng cách hoặc tạo list.**
    *   **Nguy hiểm:** `<br>` chỉ dành cho ngắt dòng thơ. Để tạo list hoặc layout, phải dùng thẻ `<ul>`, `<ol>`, `<div>` và CSS.

5.  **Đặt nội dung ẩn (display: none) để nhồi nhít từ khóa (Keyword Stuffing).**
    *   **Nguy hiểm:** Bị Google phạt nặng (Hidden Text Penalty), loại khỏi chỉ mục tìm kiếm.

6.  **Đường URL dạng tham số rắc rối.**
    *   **Nguy hiểm:** `example.com/product?id=123&cat=5` khó đọc, khó chia sẻ, bot không hiểu ngữ cảnh bằng `example.com/product/ao-thun-nam`.

## SECTION 6: MASTER CHEAT SHEET

### Quy tắc nhanh (Quick Reference)
*   **Title:** < 60 ký tự, unique, có từ khóa chính.
*   **Description:** 140 - 160 ký tự, hấp dẫn, tóm tắt nội dung.
*   **Heading:** 1 `<h1>` / trang. Tuân thủ thứ bậc H1 -> H2 -> H3.
*   **Links:** Tránh "Click here". Dùng anchor text mô tả (ví dụ: "Tìm hiểu về HTML").
*   **Images:** Luôn có `alt` attribute. Đặt tên file rõ ràng (coffee-hanoi.jpg).

### Logic ra quyết định (If-Else)
*   **Nếu** nội dung là phần quan trọng nhất trang web: **Dùng `<main>`**.
*   **Nếu** nội dung là nhóm liên kết điều hướng: **Dùng `<nav>`**.
*   **Nếu** nội dung là một bài báo/bài viết độc立: **Dùng `<article>`**.
*   **Nếu** bạn muốn Google hiển thị Rich Snippet: **Bổ sung JSON-LD**.
*   **Nếu** chỉ để chỉnh sửa màu sắc/kích thước: **Dùng CSS hoặc `<div>`**.

### Top 5 Things to Remember
1.  **Semantic First:** Chọn đúng thẻ HTML trước khi viết CSS.
2.  **Unique Title:** Mỗi trang một Title.
3.  **Meta Description:** Quảng cáo free trên Google.
4.  **Alt Text:** Mắt cho người khiếm thị và bot.
5.  **JSON-LD:** Chìa khóa cho Rich Results.

## SECTION 1: CORE MENTAL MODEL

### Core Definitions

**Form Structure (Cấu trúc biểu mẫu)**
- Collection of input elements designed to collect user data and submit it to a server
- Reality: Acts as a data contract between client and server

**Semantic HTML (HTML ngữ nghĩa)**
- Using HTML elements for their intended purpose to convey meaning
- Why better: Improves accessibility (truy cập), SEO, and maintainability vs. generic `<div>` soup

**DOM Manipulation (Thao tác DOM)**
- Programmatically modifying the Document Object Model to update UI
- Key mental model: Browser parses HTML → builds DOM tree → JavaScript can traverse and modify it

## SECTION 2: DECISION TABLES (VERY IMPORTANT)

| Use case (Tình huống sử dụng) | Should use (Nên dùng gì) | Why (Tại sao) | Common mistake (Sai lầm thường gặp) |
|-------------------------------|--------------------------|---------------|-------------------------------------|
| Collect user text input | `<input type="text">` or `<textarea>` | Native browser validation and mobile keyboard optimization | Using `<div contenteditable>` - loses form submission, validation |
| Submit data to server | `<form>` with `action` & `method` | Handles serialization, validation, submission automatically | Manual fetch/AJAX without form - reinvents the wheel |
| Password input | `<input type="password">` | Masks characters, secure against shoulder surfing | Using `type="text"` - security risk |
| Required field | `required` attribute | Native validation prevents submission | Custom JS validation only - bypassable |
| Label-input association | `<label for="id">` | Clicking label focuses input (accessibility) | Missing labels or using placeholder only |
| Search page layout | `<form>` + `<input type="search">` | Semantic meaning, can include search button/submit | Using generic text input - no special treatment |
| Registration form validation | Client-side + Server-side validation | Defense in depth: UX + Security | Client-side only - can be bypassed |
| Grouping related fields | `<fieldset>` + `<legend>` | Semantic grouping for screen readers | Div soup - no semantic meaning |
| Multiple phone numbers | `<input type="tel" multiple>` or dynamic inputs | Mobile users get numeric keypad | Using text input - poor UX |

## SECTION 3: ARCHITECTURE & RELATIONSHIPS

```
[User Input] → [HTML Form] → [Browser Validation] → [Submit Event]
    ↓
[DOM Elements] ← [Event Handlers] ← [JavaScript]
    ↓
[FormData] → [HTTP Request] → [Server Processing]
    ↓
[Response] → [UI Update / Error Display]
```

**Hierarchy Flow:**
1. **Form Container**: Wraps all inputs, defines submission endpoint
2. **Field Groups**: `<fieldset>` for logical sections (account, personal, contact)
3. **Input Elements**: Individual data collection points
4. **Labels**: Links to inputs via `for` attribute
5. **Validation**: HTML5 attributes (`required`, `pattern`, `minlength`)
6. **Submit Handler**: Intercepts and validates before sending

## SECTION 4: CODE PATTERNS (READY TO USE)

### Pattern 1: Registration Form Structure

**Khi nào dùng**: Building user registration pages with multiple data fields

**Tại sao đúng**: Uses semantic HTML, accessible labels, and built-in validation for better UX and security

```html
<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <title>Registration Form</title>
</head>
<body>
    <form action="/register" method="POST">
        <!-- Account Section -->
        <fieldset>
            <legend>Account Information</legend>
            
            <label for="email">Email:</label>
            <input type="email" id="email" name="email" required 
                   placeholder="user@example.com">
            
            <label for="password">Password:</label>
            <input type="password" id="password" name="password" required 
                   minlength="8" placeholder="Min 8 characters">
            
            <label for="confirm">Confirm Password:</label>
            <input type="password" id="confirm" name="confirm_password" required>
        </fieldset>

        <!-- Personal Section -->
        <fieldset>
            <legend>Personal Information</legend>
            
            <label for="fullname">Full Name:</label>
            <input type="text" id="fullname" name="fullname" required 
                   pattern="[A-Za-z\s]+" placeholder="John Doe">
            
            <label for="dob">Date of Birth:</label>
            <input type="date" id="dob" name="dob" required>
            
            <label for="gender">Gender:</label>
            <select id="gender" name="gender" required>
                <option value="">Select...</option>
                <option value="male">Male</option>
                <option value="female">Female</option>
                <option value="other">Other</option>
            </select>
        </fieldset>

        <!-- Terms -->
        <label>
            <input type="checkbox" name="terms" required>
            I agree to Terms & Conditions
        </label>

        <button type="submit">Register</button>
    </form>

        // Client-side validation
```

### Pattern 3: Dynamic Form Validation

**Khi nào dùng**: When you need real-time feedback as user types

**Tại sao đúng**: Provides immediate feedback, reduces server load, improves user experience

```html
<form id="registrationForm">
    <label for="username">Username:</label>
    <input type="text" id="username" name="username" required>
    <span id="usernameError" style="color: red;"></span>

    <label for="email">Email:</label>
    <input type="email" id="email" name="email" required>
    <span id="emailError" style="color: red;"></span>

    <button type="submit">Submit</button>
</form>

<script>
    const form = document.getElementById('registrationForm');
    const usernameInput = document.getElementById('username');
    const emailInput = document.getElementById('email');
    const usernameError = document.getElementById('usernameError');
    const emailError = document.getElementById('emailError');

    // Real-time validation
    usernameInput.addEventListener('blur', function() {
        const value = this.value.trim();
        if (value.length < 3) {
            usernameError.textContent = 'Username must be at least 3 characters';
        } else if (!/^[a-zA-Z0-9_]+$/.test(value)) {
            usernameError.textContent = 'Only letters, numbers, and underscores allowed';
        } else {
            usernameError.textContent = '';
        }
    });

    emailInput.addEventListener('blur', function() {
        const value = this.value.trim();
        const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
        if (!emailRegex.test(value)) {
            emailError.textContent = 'Invalid email format';
        } else {
            emailError.textContent = '';
        }
    });

    // Final validation on submit
    form.addEventListener('submit', function(e) {
        const hasErrors = usernameError.textContent || emailError.textContent;
        const isEmpty = !usernameInput.value.trim() || !emailInput.value.trim();
        
        if (hasErrors || isEmpty) {
            e.preventDefault();
            alert('Please fix all errors before submitting');
        }
    });
</script>
```

## SECTION 5: ANTI-PATTERNS & WARNINGS

### Dangerous Practices

**1. Missing Form Attributes**
```html
<!-- WRONG: Form without action/method -->
<form>
    <input type="email" name="email">
    <button>Submit</button>
</form>
```
**Why dangerous**: Browser uses current URL, may not handle submission correctly. **Always specify `action` and `method`**.

**2. Unlabeled Inputs**
```html
<!-- WRONG: No label association -->
<input type="text" placeholder="Enter email">
```
**Why dangerous**: Accessibility nightmare - screen readers can't announce purpose. **Always use `<label for="id">`**.

**3. Client-side Only Validation**
```html
<!-- WRONG: Relying only on JavaScript -->
<input type="text" id="email">
<script>
    if (!email.includes('@')) { /* block */ }
</script>
```
**Why dangerous**: Can be bypassed by disabling JS or using tools like Postman. **Always validate on server**.

**4. Password in URL**
```html
<!-- WRONG: GET method for sensitive data -->
<form action="/login" method="GET">
    <input type="password" name="pwd">
</form>
```
**Why dangerous**: Password appears in browser history, server logs. **Always use POST for sensitive data**.

**5. Radio Without Name**
```html
<!-- WRONG: Radio buttons not grouped -->
<input type="radio" value="male">
<input type="radio" value="female">
```
**Why dangerous**: Won't function as mutually exclusive selector. **Same `name` attribute required**.

**6. Wrong Input Types**
```html
<!-- WRONG: Using text for everything -->
<input type="text" name="email">
<input type="text" name="age">
```
**Why dangerous**: No validation, wrong keyboards on mobile. **Use specific types: `email`, `number`, `tel`, `date`**.

**7. Placeholder as Label**
```html
<!-- WRONG: Using placeholder as primary label -->
<input type="text" placeholder="Email Address">
```
**Why dangerous**: Disappears on focus, no persistent label. **Use `<label>` element**.

## SECTION 6: MASTER CHEAT SHEET

### Quick Reference Rules

**Form Creation Checklist:**
- `action` = Server endpoint URL
- `method` = "GET" (search) or "POST" (registration)
- `type="email/password/tel/date"` for specialized inputs
- `required` attribute for mandatory fields
- `minlength`, `maxlength`, `pattern` for constraints
- `<label for="id">` for every input
- `name` attribute on ALL inputs (required for submission)
- Radio buttons share same `name` attribute
- `type="submit"` on action buttons

### Decision Logic (If-Else Style)

```
IF collecting data → Use <form>
IF submitting to server → POST (sensitive), GET (search)
IF user must fill field → Add required attribute
IF showing error → Use <span> near input
IF password field → type="password" + POST method
IF search functionality → type="search" + GET method
IF multiple choice → Use radio buttons (same name) or <select>
IF grouping fields → Use <fieldset> + <legend>
IF validating input → HTML5 attrs + JavaScript blur event
```

### Top 10 Things to Remember

1. **Always** use `<label for="id">` - accessibility is non-negotiable
2. **Never** trust client-side validation alone - server must validate
3. **Always** specify `action` and `method` on `<form>` elements
4. **Use** `type="password"` for passwords, `POST` for submission
5. **Always** include `name` attribute on inputs or they won't submit
6. **Use** semantic HTML: `<fieldset>`, `<legend>` for grouping
7. **Validate** on `blur` event for real-time feedback
8. **Check** `required` attributes before allowing submission
9. **Use** appropriate input types for mobile keyboards
10. **Remember**: Registration = POST, Search = GET
